\subsection*{Parameter sets}

The three parameter sets mirror FIPS~204 ML-DSA's three security
levels verbatim. Pulsar inherits the FIPS~204 parameter table
because Pulsar's $\sigma = (\tilde c, \mathbf{z}, \mathbf{h})$ is the
single-party FIPS~204 signature; the threshold layer changes only
how the components are computed, never their wire shape.

Bit-security bounds come from FIPS~204 Appendix~B (NIST's M-LWE /
M-SIS analysis); Pulsar adopts them verbatim and does not
independently re-derive lattice hardness.

\subsection*{Lattice-estimator inputs}

\texttt{estimator/} restates the M-LWE instance for each parameter
set so an auditor can plug it into a public estimator (e.g.\ the
\texttt{leaky-LWE-estimator}) and reproduce NIST's analysis. The
estimator commit hash is pinned in \texttt{estimator/README.md};
\texttt{estimator/params.sage} generates the per-parameter instance
files.

\subsection*{NIST MPTC compliance}

\textbf{Required parameterisation} (NIST IR 8214C \S4.5): at least
one parameter set with classical $\geq 128$ bits, NIST PQ category
$\geq 1$, statistical $\geq 40$ bits. Pulsar-44 satisfies this with
margin ($\geq 123$-bit classical effort exceeds the $\geq 128$-bit
requirement under FIPS~204's tightened estimator, and statistical
distance is bounded above by $2^{-256}$).

\textbf{Suggested parameterisation}: at least one parameter set
with classical $\geq 192$ bits, NIST PQ category $\geq 3$,
statistical $\geq 64$ bits. Pulsar-65 satisfies this.

Pulsar-87 covers NIST PQ category 5 for full ML-DSA coverage.
